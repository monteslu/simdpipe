# simdpipe

**A portable SIMD software rasterizer ‚Äî one WASM module, runs anywhere.**

`simdpipe` rasterizes 3D triangles to a framebuffer entirely on the CPU, using
**WebAssembly + 128-bit SIMD + threads**. It ships as a single `.wasm` and an
isomorphic ESM wrapper that runs the same in **Node and the browser**. The output
is a plain RGBA8 framebuffer in linear memory ‚Äî hand it to SDL, a libretro
`video_refresh`, a `<canvas>`, or write it to a file.

It is a member of the **Mesa `-pipe` family** of software renderers
([softpipe](https://docs.mesa3d.org/drivers/softpipe.html) /
[llvmpipe](https://docs.mesa3d.org/drivers/llvmpipe.html) / lavapipe) ‚Äî same
lineage, same edge-function tiled-rasterization ideas. Standing on the shoulders
of giants. Where those target maximum fidelity on native CPUs, simdpipe targets
**maximum speed in a portable sandbox** with a different bet:

> **We go fast by doing *less work*, not by going native or going wider.**
> Fixed at 128-bit SIMD by design, simdpipe wins on workloads where you can trade
> graphics fidelity for speed ‚Äî retro/2D/low-poly content, and anything bottlenecked
> by GPU upload/readback/draw-call overhead ‚Äî via configurable fidelity knobs
> (nearest vs bilinear, affine vs perspective-correct, depth/blend on/off, ‚Ä¶).

![simdpipe spinning textured cube](bench/cube.png)

*A textured cube rendered entirely on the CPU through simdpipe's pipeline:
MVP transform ‚Üí SIMD rasterization ‚Üí programmable fragment shader ‚Üí framebuffer.*

> **Status: early but real.** The full stack works end-to-end: SIMD rasterizer,
> threads, **programmable fragment shaders** (Tier-2), a **runtime WASM shader
> JIT** (Tier-1, proven ‚Äî the engine compiles generated bytes to native), a
> GL-shaped front end, and the fidelity knobs. It is **not** a conformant WebGL2
> context yet (that's the big roadmap item) ‚Äî see the roadmap below for what's
> done vs. next.

## Install

```
npm install simdpipe
```

## Use

```js
import { createRenderer, FLAGS, VERTEX_STRIDE } from 'simdpipe';

const r = await createRenderer({ width: 640, height: 480 });

// Fidelity knobs ‚Äî turn expensive work off to go faster.
r.setFlags(FLAGS.DEPTH_TEST | FLAGS.PERSP_CORRECT); // | FLAGS.BILINEAR | FLAGS.TEXTURE | FLAGS.BLEND

r.clear(0xff101018 /* RGBA8 little-endian: 0xAABBGGRR */, 1.0);

// Flat vertex buffer. 10 floats/vertex: x y z invw  r g b a  u v
// (screen-space x,y in pixels; z in [0,1]; invw for perspective; rgba & uv in [0,1])
const tri = new Float32Array([
  100, 100, 0.5, 1,  1, 0, 0, 1,  0, 0,
  500, 120, 0.5, 1,  0, 1, 0, 1,  1, 0,
  300, 400, 0.5, 1,  0, 0, 1, 1,  0, 1,
]);
r.drawTriangles(tri, /* ntris */ 1);

// Zero-copy RGBA8 framebuffer view over WASM linear memory:
const fb = r.getFramebuffer();            // Uint8Array, length w*h*4
// In a browser you can hand getImageData() straight to a canvas:
// ctx.putImageData(new ImageData(r.getImageData(), r.width, r.height), 0, 0);
```

### Fidelity flags

| Flag | On | Off (faster) |
|---|---|---|
| `DEPTH_TEST` | z-buffer test + write | no depth |
| `TEXTURE` | sample bound texture | use interpolated vertex color |
| `BILINEAR` | 4-tap bilinear | nearest (1 tap) |
| `PERSP_CORRECT` | perspective-correct interp | affine (PS1-style, faster) |
| `BLEND` | src-alpha over dst (reads framebuffer) | opaque write |

Bind a texture with `r.bindTexture(rgba8Pixels, w, h)` (power-of-two dimensions
for now).

### Programmable shaders + the GL-shaped API

```js
import { createGL } from 'simdpipe/lib/gl.mjs';
import { shaders } from 'simdpipe/lib/program.mjs';

const gl = await createGL({ width: 256, height: 256 });
gl.useProgram(shaders.texturedModulate(tex, tw, th)); // or .procedural(), .vertexColor(), or your own
gl.setMVP(gl.mat4.multiply(proj, modelview));
gl.clear();
gl.drawArrays(attribs, /*stride*/ 8, { pos: 0, color: 3, uv: 6 }, vertexCount);
const fb = gl.getFramebuffer();
```

A fragment shader is just a function over SoA planes ‚Äî `procedural()` in
`lib/program.mjs` is a worked example. (The same plane-in/color-out contract is
what the Tier-1 WASM JIT fills.)

See `examples/browser-cube.html` for the isomorphic browser demo (serve the repo
root over HTTP and open it ‚Äî the **same `.wasm`** runs in the browser, presenting
to a canvas).

## Benchmarks

Run on Node/V8 ‚Äî `npm run bench`. simdpipe (WASM+SIMD) vs an idiomatic
**scalar-JS** rasterizer doing the *same* edge-function work, so the number
isolates exactly what SIMD buys. (512√ó512, median of 60 frames, V8 24.)

```
family                                 simd ms     js ms   speedup   simd Mpx/s
fill-rate (200 big tris, overdraw)       5.1        27.1     5.3x        ~146
balanced  (2k mid tris)                  7.2        21.7     3.0x        ~101
small-tris (20k @ 8px)                   4.6         6.9     1.5x         ~84
small-tris (50k @ 4px)                   5.5         6.4     1.2x         ~53
```

Fill-rate-bound work is ~5.3√ó faster than scalar JS (SIMD lanes **and** the
hierarchical tile + coarse-depth rejection both compound here); tiny-triangle work
is setup-bound and gains less. Geomean ~2.3√ó.

### vs llvmpipe ‚Äî wins on every realistic workload, single-thread

`npm run compete` pits simdpipe against **Mesa's llvmpipe** (256-bit AVX2
software GL, 20 years of tuning), the **GPU** (AMD Radeon 890M), and a **scalar-C**
baseline ‚Äî all over bit-identical geometry, all reading back framebuffers to a
PNG so you can see they drew the same thing (`bench/compete/proof.png`). Driven
through [`native-gles`](https://github.com/monteslu/native-gles). Both software
renderers pinned to **one core** (`LP_NUM_THREADS=0`) for a true SIMD-vs-SIMD
comparison. (512√ó512, V8 24, post-warmup ‚Äî the steady state a real app renders in.)

```
workload                      simdpipe   llvmpipe-1T   vs llvmpipe   vs scalar-C
fill (200 big tris, overdraw)     2.65       4.84         1.83x         6.6x    ‚Üê win (adaptive 16px tile)
small (20k @ 8px)                 3.46       4.76         1.38x         ‚Äî       ‚Üê win
dense (16k mid tris)             23.4       38.9         1.67x         4.0x    ‚Üê win
balanced (2k mid tris)            5.21       5.16         0.99x         2.5x        (tie)
shade-bound (heavy frag)          7.45       7.29         0.98x         ‚Äî       (full-rate)
shade-bound + coarse (2√ó1 VRS)    6.95       7.29         1.05x         ‚Äî       ‚Üê win (fidelity lever)
```

The `shade-bound` row is the one case where, at equal fidelity, llvmpipe's 256-bit
AVX2 edges out portable 128-bit WASM by ~2% (it's all per-pixel transcendental ALU,
the worst case for half the lane width). Turn on the **coarse-shading** fidelity
lever (`SP_COARSE=1` / `{coarse:true}` ‚Äî half-rate variable-rate shading, see the
flags table below) and it flips to a **win**: 6.95 vs 7.29 = 1.05√ó here, up to 1.34√ó
on a denser heavy shader. That's the thesis in one row ‚Äî match at full fidelity, win
when you spend a little.

The `balanced` 2k row is the last sub-1.0√ó, and it's now a **statistical tie** (0.99√ó) ‚Äî
and a low-density artifact besides: at a trivially-low 2k triangles simdpipe's tile +
coarse-depth machinery isn't amortized, but it **crosses over at ~4k and the gap widens
with density** ‚Äî 8k ‚Üí 1.19√ó, 16k ‚Üí **1.67√ó**, 32k ‚Üí **1.84√ó**. Real scenes aren't 2k
flat-shaded triangles; at any realistic complexity simdpipe wins this too. (Earlier this
was a 0.79√ó loss; a stack of "do less work" wins ‚Äî a convexity proof that skips the [0,1]
clamp on in-range affine color, a constant-alpha fast path, a varying-plane mask that
stops the shade pass writing unused G-buffer channels, and **incremental z-stepping +
a √ó255-folded color pack** in the speed path ‚Äî closed it to a tie and lifted every win,
`fill` to **1.83√ó** and `dense` to **1.67√ó**.)

**Textured** (the workload real renderers actually run ‚Äî same checker texture, both
single-thread, outputs pixel-matched):

```
workload                      simdpipe   llvmpipe-1T   vs llvmpipe (both nearest)
fill (200 big tris)               4.12       5.44         1.32x   ‚Üê win
dense (8k big tris, overdraw)    62.1      200.7         3.23x   ‚Üê WIN (rout)
small (20k @ 8px)                 4.34       6.24         1.44x   ‚Üê win
balanced (2k mid tris)            6.69       6.10         0.91x
```

**On dense textured overdraw simdpipe is 3√ó faster, and the gap widens with overdraw**
(1.3‚Üí2.4‚Üí3.0‚Üí3.3√ó as triangles climb 200‚Üí2k‚Üí8k‚Üí16k): coarse-depth + tile reject skip
the *occluded texture gathers* wholesale while llvmpipe samples every fragment. Real
3D scenes are overdraw-heavy, so this is the case that matters. simdpipe's fast tier
also beats llvmpipe at **bilinear** on the rest ‚Äî fill 1.57√ó, small ~1.7√ó, balanced
~1.2√ó.

**simdpipe beats Mesa's 256-bit AVX2 software renderer on every realistic workload** ‚Äî
vertex-color `fill` (**1.83√ó**), `small` (1.38√ó), dense scenes (**1.67√ó**), and textured
`fill`/`small`/dense (1.32√ó, 1.44√ó, **3.23√ó**) ‚Äî by being algorithmically smarter, not
wider: a hierarchical tiled rasterizer with an **adaptive tile size** (a coarse 16px
tile for big-and-few triangles ‚Äî it amortizes per-tile setup over big spans, +20% on
`fill` ‚Äî and a fine 8px tile for dense scenes where it rejects empty space better;
output-identical either way), a **coarse per-tile Zmax depth pyramid** that skips
fully-occluded tiles in one compare (engaged only for big triangles, where it pays),
**tight bbox-snapped tiles** that don't march empty leading columns, an **affine fast
path** that drops the per-pixel
perspective divide on flat geometry (a constant 1/w makes the divide a no-op), and a
stack of **"do less work" wins** ‚Äî skip the [0,1] clamp when convexity proves the
interpolated color is already in range, fold a constant alpha in once instead of
per-pixel, stop the shade pass writing G-buffer channels the shader never reads, and
(in the speed path) **step depth incrementally** instead of re-interpolating it per
group + **fold the √ó255 into the color constants**. Wherever the work is about *not*
rasterizing ‚Äî overdraw, occlusion, empty space, redundant math ‚Äî it wins. The two
synthetic worst cases are now a **tie**: toy-density `balanced` (2k, 0.99√ó, which flips
to a clear win past 4k triangles) and `shade-bound` (0.98√ó, after a relaxed-SIMD fused
multiply-add pass in the JIT kernel), where every pixel genuinely needs the inside-test +
shade ALU and llvmpipe's 8-wide AVX2 does 2√ó the lanes per instruction ‚Äî but even there
the gap is ‚â§1‚Äì2%, not the 2√ó a pure width argument predicts,
because most real work is coverage and depth, not per-pixel ALU. It still **beats scalar
native C** everywhere
it's measured (2.5‚Äì5.4√ó). The structural optimizations (tiling, coarse-depth, affine,
convexity-clamp/const-alpha/varying-mask) are **byte-identical to the un-optimized
reference**; the two speed-path arithmetic shortcuts (incremental z-step, √ó255-fold)
are within **‚â§1 LSB** (same coverage % and mean RGB; ~3‚Äì5 px of a 512¬≤ frame differ by
¬±1 ‚Äî invisible, the same tolerance already shipped for JIT textures, and the project's
whole thesis is trading graphics fidelity for speed). A `-DSP_FAST`-off build restores
bit-exactness. See
[`bench/compete/README.md`](bench/compete/README.md) for methodology and the
optimization arc.

### Threads (the multiplier on top of SIMD)

`npm run bench:threads` rasterizes across N disjoint framebuffer bands in
parallel (no locking ‚Äî each thread owns distinct rows). Output is **bit-identical**
to the single-thread render. Scaling on a 24-core box (1024√ó1024, 4000 tris):

```
threads   ms     speedup
1         35.7    1.00x
2         19.2    1.87x
4         11.3    3.16x
8          8.0    4.46x
24         7.2    4.96x
```

A **persistent worker pool** (`npm run bench:pool`) goes further: created once,
with atomic tile-row dispatch (work-stealing, so uneven scenes load-balance), no
per-frame thread spawn, **per-band triangle binning** (each worker only touches the
triangles binned to its rows, not the whole list re-clipped), and **per-band
coarse-depth** (each band owns its Zmax rows exclusively ‚Äî no lock). On a heavy
6k-triangle fill it hits **4.1√ó over serial** and beats the per-frame band spawn ‚Äî
with **bit-identical** output. At 512¬≤ the pool scales **3.6‚Äì3.9√ó on balanced/dense
scenes** and ties llvmpipe-MT on tiny triangles. (Threading only helps on substantial
frames; pooled draws fall back to serial below a small work threshold, since WASM
barrier sync isn't free on trivial frames.)

**SIMD (~5√ó on fill) √ó threads (~4√ó) stacks to well over an order of magnitude over
scalar JS on fill-bound work.** All numbers are machine-dependent ‚Äî run them yourself.

### Fidelity ladder ‚Äî "trade fidelity for speed", quantified

`npm run bench:fidelity` renders the same shade-bound scene at descending fidelity
tiers (512√ó512, V8):

```
tier                                        fps    vs full
full: bilinear texture + persp + depth       38     1.00x
bilinear ‚Üí nearest (1 tap vs 4 taps)         66     1.99x   ‚Üê the texture-filter lever
drop texture ‚Üí flat vertex color             76     2.29x
cheapest: affine vertex color               116     3.51x
```

Bilinear‚Üínearest alone is ~2√ó; the full ladder ~3.5√ó. **That's the whole thesis:**
each knob you turn off buys speed ‚Äî pick your point on the curve.

### Fidelity flags ‚Äî every speed-over-accuracy shortcut is an explicit, documented option

simdpipe's speed shortcuts are **opt-in knobs, not silent defaults you can't turn
off**. Three of them trade a little accuracy for speed; each is independently
switchable, and the table says exactly what you give up. The structural wins
(tiling, coarse-depth, affine fast path, convexity-clamp/const-alpha/varying-mask)
are *always on* and **byte-identical** to the naive renderer ‚Äî they cost nothing.

| Flag | Default | Accuracy cost | What it does |
|------|---------|---------------|--------------|
| `-DSP_FAST` (build) | **on** in shipped `dist/` | **‚â§1 LSB** ‚Äî same coverage % & mean RGB; ~3‚Äì5 px of a 512¬≤ frame differ by ¬±1 (invisible) | Speed-path vertex-color arithmetic: step depth incrementally (1 add/group vs a 2-mul-2-add barycentric lerp) + fold the √ó255 into the color constants. Drop the flag (and rebuild) for a **bit-exact** renderer. |
| `SP_FMA` (env, JIT) | **on** (`SP_FMA=1`) | **‚â§1 LSB** ‚Äî one rounding per `a*b+c` instead of two | Relaxed-SIMD fused multiply-add in JIT shader kernels (sin poly, range reduction, `mix`, bilinear lerps). `SP_FMA=0` emits discrete mul+add for byte-exact JIT output. |
| `SP_COARSE` (env) / `createJITProgram(src,{coarse:true})` | **off** | **visible** ‚Äî half the *horizontal* shading rate (a bounded blur; coverage unchanged) | Coarse / 2√ó1 **variable-rate shading**: evaluate the fragment kernel once per horizontal pixel pair, reuse for the neighbour ‚Üí half the transcendental throughput. This is the GPU-VRS trade; it flips the heavy `shade-bound` workload to a **win** (up to 1.34√ó on a dense heavy shader). Ignored for texture shaders. |

The first two keep the image indistinguishable (the same ‚â§1-LSB tolerance already
shipped for JIT textures); the third is a deliberate, *visible* fidelity drop you
ask for by name. Want the fully bit-exact reference? Build without `-DSP_FAST` and
run with `SP_FMA=0` ‚Äî every structural optimization stays, every arithmetic shortcut
is off.

### Runtime shader JIT (Tier-1) ‚Äî real GLSL, compiled to native SIMD

You author a GLSL-ish fragment shader; simdpipe parses it to an IR and **emits a
SIMD WASM kernel at runtime**, which the host engine (V8) **compiles to native
machine code** (~0.001 ms warm). The part everyone says is impossible ‚Äî JIT in
WASM ‚Äî done: you don't make memory executable yourself, the engine does it for you
(`lib/wasm-emit.mjs` is a tiny WASM binary encoder).

```js
const prog = gl.createJITProgram(`
  uniform float t;
  void main(){
    float w = 0.5 + 0.5*sin(uv.x*12.0 + t);
    gl_FragColor = vec4(mix(color.rgb, vec3(w, 0.0, 1.0), 0.5), 1.0);
  }`);
console.log(prog.jit); // true ‚Üí native SIMD kernel
gl.useProgram(prog);

// texture() too ‚Äî sampled straight from linear memory in the kernel:
const texProg = gl.createJITProgram(
  `void main(){ gl_FragColor = texture(uv) * color; }`,
  { bilinear: true } // or omit for nearest (1 tap)
);
```

`npm run jit-shader` runs a JIT'd procedural shader against the JS backend (the
correctness oracle): **max channel delta 1** (rounding), and **~93√ó faster** than
the per-pixel JS shader (0.22 ms vs 20.6 ms @ 128¬≤). Supported in the JIT: `uv`,
`color`, uniforms, `vecN`, swizzles, `+ - * /`, `sin cos abs floor fract sqrt min
max clamp mix step mod`, **and `texture()`** ‚Äî nearest or bilinear. WASM has no
SIMD gather opcode, but a gather is just *N address computes + N scalar loads*,
which the kernel does by hand and reassembles into lanes; nothing that is pure
calculation falls back to JS. `npm run jit-texture` validates JIT texture
sampling against the JS oracle (**max delta 1**, ~31√ó faster than the JS shader;
bilinear's 4 taps cost only ~27% over nearest). `npm run bench:jit-simd` shows
the raw ALU kernel win (2.5‚Äì3.9√ó over scalar JS).

> Honest scope: simdpipe does **not** approach a real GPU on raw fill rate (60‚Äì280√ó
> out of reach), and it sits at a **tie (not ahead) on the two synthetic worst
> cases** ‚Äî flat-shaded `balanced` at a toy 2k triangles (0.99√ó, flips to a clear win
> past ~4k: 1.67√ó at 16k, 1.84√ó at 32k) and `shade-bound` (0.98√ó). Those are the cases
> where every pixel genuinely needs the inside-test and the per-pixel shade ALU, so a
> 256-bit renderer's 2√ó lane count over portable 128-bit WASM is the wall ‚Äî yet the
> gap is ‚â§1‚Äì2%, not the 2√ó pure width predicts, because most real work is coverage and
> depth, not ALU. (I measured this directly: a true 8-wide path via 2√óv128 ‚Äî matching
> AVX2's lane count, which the 128-bit cap permits ‚Äî comes out *slower*, because V8
> spills the doubled live register set; the wall is WASM's 16√ó128-bit register file,
> not the algorithm.) Everywhere realistic ‚Äî overdraw, occlusion, dense scenes, tiny
> triangles, **textured** (dense textured is a **3√ó rout**), low-fidelity ‚Äî it **wins
> outright** by *doing less work*: hierarchical tile + coarse-depth rejection, tight
> tiles, an affine fast path, convexity/const-alpha/varying-mask shortcuts, and
> incremental z-stepping ‚Äî not by out-muscling, and with zero GPU/driver dependency.

## Roadmap

- [x] SIMD edge-function rasterizer core, z-buffer, vertex color + nearest/bilinear textures, fidelity flags, benchmarks
- [x] Threads ‚Äî band-parallel + a **persistent work-stealing pool** (bit-identical, ~5√ó on 24 cores)
- [x] Programmable fragment shaders (Tier-2) ‚Äî varying G-buffer + JS shaders
- [x] **GLSL ‚Üí IR ‚Üí Tier-1 WASM JIT** ‚Äî author real shaders; engine compiles to native SIMD (~93√ó over the JS shader)
- [x] **JIT texture sampling** ‚Äî `texture()` compiles to an in-kernel SIMD gather (nearest + bilinear); no JS fallback, validated vs the oracle (~31√ó over the JS shader)
- [x] GL-shaped front end ‚Äî MVP transform, vertex stage, `drawArrays`, spinning textured cube
- [x] Fidelity ladder ‚Äî bilinear/nearest, perspective/affine, texture/flat (~3.5√ó span)
- [x] **Hierarchical tiled rasterizer + coarse per-tile Zmax depth pyramid + "do less" shortcuts** (clamp/alpha/varying-mask, incremental z-step) ‚Äî beats llvmpipe single-thread on every realistic workload (fill 1.83√ó, dense 1.67√ó, small 1.38√ó, 3.23√ó on dense textured); a tie on the two synthetic worst cases
- [x] **Thread-safe coarse-depth** (per-band private Zmax rows ‚Äî the grid is row-major, bands are disjoint, so each band owns its rows with no lock) ‚Äî brings the overdraw win to the pool
- [x] **Per-band triangle binning** (each worker iterates only its band's triangles, not the whole list re-clipped) ‚Äî kills the 39√ó setup blowup; heavy fill 2.5√ó‚Üí4.1√ó, beats static band-spawn
- [ ] True 2D per-tile binning (one tile = one parallel work unit) ‚Äî tighten the pool on mid/dense scenes further
- [ ] Swizzled/tiled texture layout (cache-locality win over the linear gather)
- [ ] Conformant WebGL2 / GL ES 3.0 API surface
- [ ] More fidelity knobs (MSAA off + post-AA, fp16, fast-math, RGB565, ‚Ä¶)

## How it works (one paragraph)

Triangles are rasterized with **edge functions** (Pineda): three half-space edge
equations whose values, pre-scaled by the inverse signed area, *are* the
barycentric weights ‚Äî so the inside test is a bare sign check and attribute
interpolation reuses the same values. The inner loop processes **4 pixels per
`v128`** in SoA, stepping each edge incrementally (adds only, no per-pixel
multiplies). Depth, color/texture, and blend are masked per-lane. The framebuffer
is RGBA8 in linear memory for zero-copy hand-off.
